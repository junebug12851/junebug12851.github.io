# System Map — overview

The machine, end to end. Four cooperating systems live in one repo.

## 1. The website (Jekyll → Pages)

```
authored content                 GitHub Actions                 GitHub Pages
(_config, _layouts,   ──push──>  ruby/setup-ruby                served at
 _includes, pages,    to main    bundle exec jekyll build  ──>  junebug12851.github.io
 _posts, _data, css)             upload-pages-artifact          + fairyfox.io (CNAME)
                                 deploy-pages
```

- Static site, built on every push to `main`. No manual publish step.
- `_data/projects.yml` is the single registry; the home page and `/projects/`
  both render from it.
- Output is `_site/` (git-ignored). Feeds/sitemap/SEO are generated by plugins.
- Details: [`../reference/deployment.md`](../reference/deployment.md).

## 2. The blog

- Posts live in `_posts/YYYY-MM-DD-title.md`, rendered with the `post` layout,
  surfaced on the home page (latest few) and `/blog/` (all), and syndicated via
  `/feed.xml`.
- Two kinds of post: ordinary write-ups, and **cross-project round-ups** —
  generated by reading the reference clones and summarizing what changed. The
  procedure is [`../reference/blogging-workflow.md`](../reference/blogging-workflow.md).

## 3. The hub (outbound standards)

- [`../../hub/`](../../hub/) holds the project-agnostic **standards**
  (`hub/standards/`), copy-paste **templates** (`hub/templates/`), and the machine
  **registry** of projects (`hub/registry.yml`).
- Other repos pull `hub/` to adopt or refresh shared conventions. The hub is the
  source of truth for shared standards.

## 4. Cross-project sync (the wiring between repos)

Two independent, one-directional git flows — never automatic, **always on
explicit request**, both tracking the **`dev`** branch:

```
        hub reads projects                     projects read hub
  ┌─────────────────────────────┐      ┌──────────────────────────────┐
  │ fairyfox.io                  │      │ some-project                 │
  │  assets/references/<proj>  <─┼──────┤  (its dev branch)            │
  │   (clone of proj dev;       │ pull │                              │
  │    git-ignored)             │ dev  │                              │
  │                             │      │  assets/references/fairyfox  <┼── pull
  │  hub/  ─────────────────────┼──────┼─> (clone of hub dev)         │   dev
  └─────────────────────────────┘      └──────────────────────────────┘
```

- **No submodules, no live dependency, no build coupling.** Just git.
- **Pull-only + on-request** is the anti-recursion rule: nothing here triggers a
  pull in the other repo, so there's no loop.
- The inbound clones carry two things the hub reads: project **history** (for
  round-ups) and each node's **process reports** (`notes/fairyfox-reports/`) — the
  feedback loop where nodes report on each system run and the hub reviews it to
  improve the standards ([`../../hub/standards/process-reports.md`](../../hub/standards/process-reports.md)).
- Full model + commands: [`../reference/cross-project-sync.md`](../reference/cross-project-sync.md).

## How the four relate

The website is the public face; the blog is its voice; the hub is the shared
spine other repos lean on; the sync is the nervous system that lets the hub and
the projects learn about each other without being entangled.
