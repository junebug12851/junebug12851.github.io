## CI service secrets — new standard + `repo-tokens` tool (1.3.0)

Owner: three per-repo secrets (`SONAR_TOKEN`, `CODECOV_TOKEN`, `SCORECARD_TOKEN`) that every
applicable node must set, via a masked-input script that pipes straight to `gh` — "was
supposed to be in the compliance checks already." Confirmed the gap: badges / testing /
supply-chain-hardening all reference the services, but nothing named or provisioned the
secrets. Clarified with the owner: all three are **single-line tokens** (Scorecard is a PAT,
not a PEM), the tool lives in the shared hub, and it gets a real `## Verify` gate.

Built:

- `hub/tools/repo-tokens.ps1` (new `hub/tools/` dir) — prints each secret's get-it link +
  how-to, prompts **concealed** (`Read-Host -AsSecureString`), streams the value to
  `gh secret set … --body-file -` over stdin and zeroes the plaintext (no disk / history /
  args / transcript). Blank = skip; `-Repo` override; `-Only` subset; `gh secret list` at the
  end. Parse-checked, CRLF-normalized (per `.gitattributes`); `gh` preflight verified.
- `hub/standards/ci-secrets.md` — the standard (secrets table + links, the tool, applicability
  filter, `## Verify`). Row added to `compliance.md`; `VERIFY-INDEX.md` regenerated (28
  standards). `CLAUDE.md` documents the **"fairyfox repo tokens"** trigger.

MINOR (new shared capability). Build + link gate green. Releasing PR-based (main is now
protected). VERSION `1.2.0 → 1.3.0`.
