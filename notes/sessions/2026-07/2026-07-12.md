### Two git findings, worth recording

**1. `main` could not be fast-forwarded, and that was correct.** The release step aborted:
`main` carries `--no-ff` "Release vX.Y.Z" merge commits (v0.15.3 … v0.15.8) that had **never been
back-merged into `dev`**, so the branches had diverged and `merge --ff-only dev` was impossible.
Nothing was forced. The project's own `CLAUDE.md` prescribes the right path for a PATCH —
`git merge --no-ff dev` + an annotated tag — so that is what was used (`v0.15.9`), and `dev` was
then fast-forwarded up to `main`, which **restores the dev-contains-main invariant** the
git-workflow standard has required since `0.12.0`. Worth knowing: the invariant had been quietly
broken for five releases, and a plain `--ff-only` release will keep failing until each release
back-merges.

**2. The git remote still pointed at the old owner.** It worked only on GitHub's redirect
(`remote: This repository moved`). Re-pointed to `git@github.com:1fairyfox/junebug12851.github.io.git`
(local config, not committed).
