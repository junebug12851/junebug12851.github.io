### 2026-06-27 — Standards: kill the phantom "force-push" — full-history mirrors, no routine `reset --hard` (0.9.6)

Root-cause fix for a recurring complaint. The sync standards claimed the hub's `dev`
"is force-pushed routinely" and told every mirror refresh to `pull --depth 1 --ff-only`
and, when that aborted, to `reset --hard origin/dev`. **None of that was true or
necessary.** `dev` is append-only everywhere in the mesh — nothing force-pushes it (a
hard safety rule in `git-workflow.md`). The real cause of the aborts was the
**`--depth 1` shallow clone itself**: a shallow mirror has no merge base, so
`merge --ff-only` fails with `refusing to merge unrelated histories` even on a clean
fast-forward — and the procedure misread that as a force-push and reached for a
destructive `reset --hard`. Proven directly: after `git fetch --unshallow`, the same
refresh fast-forwards cleanly and the old tip is confirmed a true ancestor of the new.

- **The fix:** reference mirrors are now **single-branch, full-history** clones
  (`clone --branch dev --single-branch`, *not* `--depth 1`), and the refresh is a plain
  `git fetch` + `git merge --ff-only`. If `--ff-only` ever aborts it is treated as an
  **anomaly to diagnose** (almost always a leftover shallow mirror → `git fetch
  --unshallow` or re-clone), never a routine to `reset --hard` through. A genuine
  non-fast-forward means someone rewrote `dev` — stop and investigate.
- **Files:** `hub/standards/cross-project-sync.md`, `adopting-updates.md` (step 1 refresh,
  step 2 anchor, close-out, Verify), `process-reports.md`, `new-project-setup.md`,
  `onboarding-existing-project.md`, `compliance.md`; template `fairyfox-report.md`;
  `hub/README.md`, `hub/templates/README.md`, `assets/references/README.md`;
  `hub/.last-seen.yml` header; and the hub's adopted `notes/reference/cross-project-sync.md`.
  All `--depth 1` / `reset --hard` / "force-pushed routinely" wording removed or corrected.
- **Local mirrors** under `assets/references/` were unshallowed; both now fast-forward
  cleanly. Nodes pick this up through ordinary adoption.

### 2026-06-26 — Standards tightened from the siblings' process reports (0.9.1)

First real fold-back of node feedback into the standards. The inbound review pass
read the two siblings' `2026-06-26-adopting-updates.md` reports; both, independently,
hit the same two snags, so those drove the changes.

- **Force-push refresh is now first-class (`adopting-updates.md`, `cross-project-sync.md`).**
  The hub's `dev` is force-pushed routinely, so the documented `--ff-only` refresh
  aborts every time. Both nodes had to improvise. Step 1 now treats the abort as
  expected and leads with the lighter `git fetch` + `git reset --hard origin/dev` on
  the **git-ignored mirror** (re-clone as the heavier fallback), with a loud guardrail
  that `reset --hard` is allowed *only* on the disposable mirror, never a tracked
  branch. Both sync flows carry the fallback inline.
- **Change detection anchors on VERSION, not a commit (`adopting-updates.md` step 2,
  `process-reports.md`).** Because force-push erases the old commit, a SHA diff is
  unreliable; both nodes reconstructed the delta by hand. The durable anchor is the
  **`hub_version` in the node's newest adopting-updates report** — read the hub's
  append-only `notes/version/` changelog across that span; a file diff is demoted to a
  noisy secondary check. No new marker file: the report front matter *is* the anchor.
- **CI-owns-tagging warning (`git-workflow.md`, `adopting-updates.md` step 5).**
  pokered-save-editor-2's `release.yml` auto-tags and skips if the tag exists, so the
  runbook's manual `git tag` would silently no-op the release. The release step is now
  a decision: if CI owns tagging, don't hand-tag.
- **Report lifecycle gaps closed (`process-reports.md`).** A check that becomes an
  adopt in one session is one combined report; a check-only run on a node that hasn't
  adopted process-reports reports inline and writes no file (creating the folder is
  itself adoption — the chicken-and-egg random-ai-prompt flagged).
- **Unprompted close-out (`adopting-updates.md`).** Every run now ends with a plain
  "what changed, where" summary (dev/main status, the `reset --hard` hit the mirror
  only), after a node owner found a bare adopt alarming.
- **Template + Verify sections** updated to match (real `hub_version` required;
  Verify clauses for the version anchor, tag ownership, close-out, and combined report).

Markers in `hub/.last-seen.yml` advanced to the digested reports. Hub-side only; the
siblings pick these up on their next adoption. Doc clarifications, no behavior change → PATCH.
