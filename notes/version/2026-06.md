### 2026-06-25 — Full git-flow is the model, not a thin core (0.6.3)

Corrected 0.6.2's framing (FF's steer): **full git-flow is the standard, adopted as much
as makes sense** — not a minimized "paper-thin core" with feature/release/hotfix as rare
expansions ("paper-thin" was his term for the *old* `dev → main` system being a thin
shadow of git-flow). Rewrote `hub/standards/git-workflow.md`: full git-flow is *the*
model — **`feature/*`** is the normal unit of work, **`release/*`** is the release
mechanism (`main` advances **only** via release/hotfix, never `dev` merged straight in),
**`hotfix/*`** for urgent prod fixes — with explicit command flows for each. Kept
canonical **`--no-ff`** throughout + tags on `main`, the `master → main` mandate, and the
safety rules. The solo simplifications (trivial change → direct on `dev`; routine release
→ direct `dev → main`) are demoted to a secondary **"Solo latitude"** note — latitude
*within* the model, not a lighter model. Mirrored into `notes/reference/git-workflow.md`;
updated `CLAUDE.md` Default Workflow steps 2–3. PATCH `0.6.2 → 0.6.3`.

### 2026-06-25 — Reframe the git standard as git-flow (0.6.2)

Mesh-wide policy (FF's call): **all projects use git-flow** — its procedures and
policies, **not** the `git flow` CLI/scripts (upheld by the AI in each repo, not
automated). Rewrote `hub/standards/git-workflow.md` around git-flow: an everyday
**paper-thin core** (commit on `dev`, release `dev → main`) plus the full git-flow
**`feature/` / `release/` / `hotfix/`** branches documented as *expansions* used only
when a change warrants them, with a `type/` branch-naming prefix. Kept the integration
branch named **`dev`** (mapped to git-flow's `develop` role — no churn to sync/registry
refs) and adopted git-flow's **`--no-ff`** merge style. Releases to `main` now use
`git merge --no-ff dev` + an annotated `vX.Y.Z` tag (matching `VERSION`) instead of a
fast-forward — **this reverses the earlier FF-only-to-`main` rule**; `--no-ff` merge
commits are additive (a release marker), not a history rewrite, so the safety rules
stand. Mirrored into `notes/reference/git-workflow.md` and updated `CLAUDE.md`'s Default
Workflow step 3 to match. PATCH `0.6.1 → 0.6.2`.

### 2026-06-25 — Docs-site design-system standard (0.5.0)

New shared standard `hub/standards/docs-site/` — a 10-file spec that lets every
project build and maintain a documentation site that's **seamless with fairyfox.io**:
same theme/colours/type/components, same structure, and required two-way linking
(project header/footer/breadcrumb back to Fairy Fox; the main site links in), all on

AI-driven projects read it and reproduce the design in their own stack (per FF's

boundaried, not faked. Files: README + 01 overview/principles · 02 design tokens
(exact dark+light palette, Fraunces/Inter/JetBrains, spacing/shadow/motion) · 03
layout/structure · 04 components · 05 navigation/cross-linking · 06
content/organization · 07 accessibility (WCAG 2.1 AA) · 08 compliance checklist · 09
adopting/maintaining. **fairyfox.io is the master copy: exempt from auto-applied
style/theme updates — manual review by FF only, one-way outward to projects.** Wired
into `hub/README.md` and a new `CLAUDE.md` maintenance trigger. MINOR `0.4.12 → 0.5.0`.
