# Process Report — roundup, 2026-06-26

> A full, honest account of running a fairyfox system procedure. The point is to
> improve the system — so say what was rough even if the run succeeded. Voice: direct,
> matter-of-fact, no hype. Standard: `hub/standards/process-reports.md`.

## Outcome in one line

Scheduled hub-side round-up: refreshed both sibling references, reconciled the data
surface and docs library to the shipped `0.7.x`/`0.8.0` releases, blogged the two
unblogged site standards, shipped `0.8.1`. First round-up to run under the
process-reports standard, so this is the first report.

## What was done

Oriented on `status.md` + the latest session log + changelog top; confirmed a clean
tree on `dev` and `main` at `v0.8.0`. Refreshed both sibling shallow clones: PSE2 was
already at its marker (`f88bbfd`); RAP's `dev` was force-pushed, so re-cloned per the
documented fallback — its only new commit (`0b64abf`) was a code-comment tidy, no
release, not blog-notable, so I advanced the marker without a post. Found real drift
from the `0.7.0`/`0.8.0` releases: `status.md`, `_data/projects.yml` (meta row), and
`_data/pulse.yml` still read `0.6.x`/`0.7.0`, and the two newest standards
(process-reports, compliance) had no docs-library page and were unblogged. Reconciled
all of it, added the two `_docs` pages, folded two sections into the existing
`2026-06-25` day post, built clean, shipped PATCH `0.8.0 → 0.8.1` via the direct
`dev → main` git-flow path (`--no-ff` + tag).

## What went well

The marker system did its job: PSE2 "already at marker" and RAP "one trivial commit
past marker" were both unambiguous calls. The re-clone fallback for the force-pushed
RAP `dev` worked exactly as the runbook describes. The data files are well-commented,
so deciding what each field should read was fast. The build was green first try and the
two new doc pages auto-appeared in the standards group with no index edit (collection +
`doc_categories.yml` drive it).

## What went wrong / friction

- **The release instruction in the scheduled-task file contradicts the project's own
  git-flow standard.** The task's step 9 says to ship a PATCH with
  `git merge --ff-only dev`, but `CLAUDE.md` + `hub/standards/git-workflow.md` (since
  `0.6.x`) mandate `--no-ff` + an annotated tag for a PATCH direct `dev → main`, and the
  `0.8.0` compliance `Verify` for git-workflow explicitly checks that `main` is
  "`--no-ff` tagged releases only." Following the task literally would have *failed our
  own compliance audit*. I followed the project standard (CLAUDE.md overrides) and
  noted it. The generic task wording predates the git-flow adoption and should be
  reconciled to match — otherwise every scheduled pass either drifts `main` history or
  has to catch this by hand.
- **Earlier releases left the data surface behind.** The `0.7.0`/`0.8.0` work updated
  the hub but not `status.md` / `projects.yml` / `pulse.yml`, and added two standards
  with no docs page or blog entry. Nothing caught this until a maintenance pass. The
  enforcement layer added per-standard `Verify` checks, but there is no check that asks
  "did a new standard get a `_docs/` page and a `projects.yml`/`status.md` version
  bump?" — i.e. the *site's own surface* isn't part of any audit.

## Suggestions / feedback

- Reconcile the scheduled fairyfox-maintenance task's release step from `--ff-only` to
  the git-flow PATCH ritual (`--no-ff` + `git tag -a vX.Y.Z` + `push --tags`), matching
  `hub/standards/git-workflow.md` and `CLAUDE.md` step 3. This is the same
  ff-only → git-flow supersession that `0.7.1` swept out of the in-repo docs; the
  external task file is the one remaining copy carrying the stale model.
- Consider a lightweight "site-surface" checklist (or a row in the compliance audit
  scoped to the hub) covering: every `hub/standards/*` has a `_docs/` page; the
  `fairyfox.io` meta row + `status.md` version line equal the newest tag. A release that
  adds a standard would then have an explicit check that the public surface kept up.
